# AGENTS.md

Короткие постоянные правила для агентов в `ai-editor`.

## Контекст

`ai-editor` — локальный desktop-редактор русского текста с autocomplete.

Стек проекта: Tauri 2, React, TypeScript, CodeMirror 6, Vitest, ESLint, Prettier, npm. Локальный inference подключается через adapters/providers и не должен протекать в UI.

## Обязательные правила

- Перед задачей использовать `docs/CONTEXT_BUDGET.md`: выбрать только релевантные документы, включая подходящий `docs/CODEMAP/*`, и кратко зафиксировать контекст.
- Если меняешь код области, обнови соответствующий файл в `docs/CODEMAP/`, чтобы карта проекта оставалась актуальной.
- Scope, ограничения и acceptance criteria брать из текущего промта задачи или отдельной feature/task spec.
- UI не вызывает напрямую Ollama, LLM, HTTP API или системные команды.
- React components не содержат бизнес-логику autocomplete, prompt-building или inference.
- Цепочка autocomplete: `Editor UI` -> `AutocompleteService` -> `SuggestionProvider` -> конкретный provider.
- Inference доступен только через adapters/providers.
- Fallback autocomplete обязателен, если задача затрагивает autocomplete.
- Перед UI-задачами читать `docs/UI_DESIGN_GUIDE.md`; при изменении визуального стиля обновлять guide или объяснять, почему это не нужно.
- Код писать на английском, документацию — на русском.
- Тесты обязательны для чистой логики.
- Не хранить секреты, токены и API-ключи в репозитории.
- Не добавлять зависимости без объяснения; необходимые системные зависимости окружения и зависимости проекта допустимы, если они нужны для запуска, сборки или тестов и указаны в отчёте.
- Изменения делать небольшими, связными и проверяемыми.
- Архитектурные решения менять только через ADR.
- MCP подключены только проектно через `.codex/config.toml`; не требовать глобальный `codex mcp add`, кроме как явную альтернативу вне этого проекта.
- Context7 использовать только когда нужна актуальная документация внешней библиотеки.
- Playwright MCP использовать только для UI/e2e-сценариев, скриншотов и проверки поведения интерфейса.
- MCP и subagents не использовать автоматически «на всякий случай».
- Subagents запускать только когда задача крупная, рискованная или явно требует workflow с subagents.
- Финальный отчёт держать коротким: изменения, проверки, ограничения, риски и затронутые файлы.

## Проверки

Доступные команды:

```bash
npm install
npm run dev
npm run build
npm run typecheck
npm run lint
npm run test
npm run check
```

Перед завершением задачи запускай доступные проверки. После feature-задачи tester должен реально проверить приложение доступными командами и, для UI, smoke/e2e-сценарием, если окружение это позволяет. Если команда отсутствует или окружение мешает запуску, не выдумывай результат: явно напиши, что не прошло и почему.
