# Task Spec Template

Шаблон для отдельной задачи. Постоянная документация задаёт правила, а конкретный scope задаётся этим документом или промтом пользователя.

## Цель

Что должно измениться и зачем.

## Контекст

- Связанные файлы или документы.
- Текущее поведение.
- Предыдущие решения, которые нельзя нарушить.

## Что реализовать

- Конкретные изменения.
- Ожидаемые состояния и сценарии.
- Нужные тесты или документационные правки.

## Что не входит в задачу

- Явные ограничения.
- Фичи, которые нельзя добавлять в этой итерации.
- Поведение, которое нельзя менять.

## Архитектурные ограничения

- Какие слои можно менять.
- Какие зависимости запрещены.
- Какие контракты нужно сохранить.

## Acceptance Criteria

- Проверяемые признаки готовности.
- Пользовательские сценарии.
- Требования к ошибкам и fallback.

## Проверки

- Команды, которые нужно запустить.
- Smoke/e2e-сценарии, если задача меняет UI.
- Что считать блокером, а что ограничением окружения.

## Subagents

- Нужен ли `architect`.
- Нужен ли `coder`.
- Нужны ли `reviewer` и `tester`.
- Когда повторять review после исправлений.

## MCP

- Нужен ли Context7 для актуальной внешней документации.
- Нужен ли Playwright для UI/e2e.
- Почему локального контекста недостаточно, если MCP используется.
