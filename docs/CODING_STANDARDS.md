# Стандарты кода

## TypeScript

- Использовать `strict` режим.
- Не использовать `any`, если тип можно описать явно.
- Предпочитать небольшие функции с понятной ответственностью.
- Использовать явные имена переменных, функций и типов.
- Предпочитать композицию лишнему ООП.
- Доменные типы держать отдельно от React-компонентов.
- Ошибки возвращать и отображать понятно.

## UI

- UI-компоненты не должны содержать бизнес-логику autocomplete.
- UI не должен напрямую вызывать Ollama, HTTP API или системные команды.
- Компоненты получают данные и callbacks через props или локальные hooks.
- Визуальные изменения должны соответствовать `docs/UI_DESIGN_GUIDE.md`.
- Сложные вычисления выносить из компонентов в сервисы или чистые функции.

## Services

- Сервисы не должны зависеть от React.
- `AutocompleteService` должен работать с интерфейсами, а не с конкретными backend.
- `PromptBuilder`, `RankingService` и providers должны тестироваться отдельно.
- Внешние эффекты изолировать за адаптерами.

## Rust

- Rust-код держать минимальным и явным.
- Нативные команды добавлять только при реальной необходимости.
- Не переносить бизнес-логику autocomplete в Rust без отдельного архитектурного решения.
- Ошибки Rust-команд возвращать в понятном для frontend формате.

## Комментарии

Комментарии нужны только там, где логика неочевидна. Не нужно комментировать очевидные присваивания, простые условия и имена, которые уже объясняют намерение.

## Документация и имена

- Кодовые имена интерфейсов, классов, функций и типов писать на английском.
- Документацию писать на русском.
- Новые архитектурные решения фиксировать через ADR.
