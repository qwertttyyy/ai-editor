# Workflow агентной разработки

Одна задача = один понятный вертикальный срез. Scope, ограничения и acceptance criteria задаёт пользовательский промт или task spec.

## Обычная задача

Для небольших правок достаточно основного агента или `coder`:

1. Прочитать `AGENTS.md` и релевантные docs по `docs/CONTEXT_BUDGET.md`.
2. Сверить scope текущей задачи.
3. Сделать минимальные изменения.
4. Запустить доступные проверки.
5. Кратко отчитаться.

## Крупная feature-задача

Рекомендуемая цепочка:

```text
architect -> coder -> reviewer -> coder, если есть замечания -> tester
```

1. `architect`
   - анализирует задачу;
   - проверяет архитектурные границы;
   - предлагает короткий план реализации;
   - не пишет код.

2. `coder`
   - реализует задачу по плану;
   - соблюдает архитектуру, coding standards и scope конкретной задачи;
   - добавляет или обновляет тесты.

3. `reviewer`
   - проверяет текущий diff;
   - ищет нарушения архитектуры, ошибки, лишнюю сложность, отсутствие тестов, несоблюдение task scope;
   - возвращает только actionable findings.

4. Если `reviewer` нашёл реальные замечания:
   - снова запускается `coder`;
   - `coder` исправляет только конкретные замечания reviewer;
   - после существенных исправлений можно повторно запустить `reviewer`.

5. `tester`
   - запускается после reviewer-этапа и исправлений;
   - проверяет тесты, acceptance criteria и основные пользовательские сценарии;
   - не переписывает код без отдельной команды.

## Типы задач

- UI task: читать `docs/UI_DESIGN_GUIDE.md`, проверять визуальные и клавиатурные сценарии.
- Testing task: читать `docs/TESTING_STRATEGY.md`, не требовать локальный inference для unit-тестов.
- Documentation task: обновлять только релевантные docs, не раздувать правила временными деталями.
- Review task: начинать с diff и scope текущей задачи.

## Subagents

Subagents не работают сами. Пользователь или основной Codex должен явно инициировать их запуск в рамках задачи.

Workflow с subagents не обязателен для каждой мелкой правки. Для крупных фич, архитектурных изменений и UI/UX-изменений он рекомендуется по умолчанию.
