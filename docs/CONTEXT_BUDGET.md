# Context Budget

Цель: тратить контекст только на то, что помогает выполнить текущую задачу.

## Главный принцип

Не читать всю документацию на каждую задачу. Сначала определить тип задачи, затем выбрать 1-3 релевантных документа. Главный источник правды — локальный код, текущий diff и файлы в scope задачи. MCP использовать только когда локального контекста недостаточно. MCP для этого проекта подключаются только проектно через `.codex/config.toml`, а не глобально через `codex mcp add`.

## Алгоритм перед задачей

1. Прочитать `AGENTS.md`.
2. Определить тип задачи.
3. Выбрать 1-3 релевантных документа.
4. Не читать остальные документы без причины.
5. Решить, нужен ли MCP.
6. Решить, нужны ли subagents.
7. Кратко зафиксировать выбранный контекст перед работой.

## MCP

MCP нельзя использовать «на всякий случай». Если задачу можно решить по локальному коду и локальным docs, MCP не использовать.

Context7 MCP подключён только проектно через `.codex/config.toml`. Использовать Context7 только если нужна актуальная документация внешней библиотеки: добавляется библиотека, нужен актуальный API, есть ошибка версии, неясна сигнатура или локальной документации недостаточно. Не использовать Context7 для внутренних задач, документации, текущей бизнес-логики и задач без зависимости от версии внешней библиотеки.

Playwright MCP подключён только проектно через `.codex/config.toml`. Использовать Playwright только для UI/e2e-сценариев, скриншотов и проверки поведения интерфейса: если меняется UI, autocomplete popup, клавиатурное поведение, тема, пользовательский сценарий, нужен скриншот или воспроизведение UI-бага. Не использовать Playwright для документации, чистой логики, типов и задач, покрываемых unit-тестами.

Не предлагать глобальный `codex mcp add` как основной способ настройки MCP для `ai-editor`. Его можно упоминать только как альтернативу для других проектов или личной глобальной конфигурации.

## Subagents

Использовать subagents только для крупных изменений, независимого review, архитектурной проверки, сложного UI/e2e сценария или перед важным коммитом. Не использовать subagents для маленьких изменений, документации, переименования, форматирования и задач, где достаточно `npm run check`.

## Отчёты

Не пересказывать документацию. Финальный отчёт должен содержать только: что изменено, какие проверки выполнены, что не удалось выполнить, какие риски остались и какие файлы затронуты.

## Зависимости

Перед добавлением зависимости объяснить: зачем она нужна, почему нельзя решить проще, есть ли альтернатива в текущем стеке и как она влияет на поддержку проекта.

## Конфликт документации и кода

Если документация расходится с текущим кодом, явно указать конфликт, не угадывать и предложить минимальный способ синхронизировать код и документацию.

## Пример context checklist

```text
Тип задачи: UI
Читаю: AGENTS.md, docs/UI_DESIGN_GUIDE.md, docs/ARCHITECTURE.md
Не читаю: ROADMAP.md, ADR, SECURITY_AND_PERMISSIONS.md
MCP: не нужен
Subagents: не нужны
```
