# Context Budget

Цель: тратить контекст только на то, что помогает выполнить текущую задачу.

## Главный принцип

Не читать всю документацию на каждую задачу. Сначала определить тип задачи, затем выбрать релевантные документы. Главный источник правды — локальный код, текущий diff и scope текущей задачи. MCP использовать только когда локального контекста недостаточно.

## Алгоритм перед задачей

1. Прочитать `AGENTS.md`.
2. Определить тип задачи.
3. Прочитать `docs/CODEMAP/README.md` и выбрать нужную карту области.
4. Выбрать остальные релевантные документы.
5. Не читать остальные документы без причины.
6. Решить, нужен ли MCP.
7. Решить, нужны ли subagents.
8. Кратко зафиксировать выбранный контекст перед работой.

## CODEMAP

`docs/CODEMAP/` — короткая карта текущего кода и функционала. Она нужна, чтобы быстро понять, какие файлы читать для конкретной задачи, без веерного просмотра всего `src/`.

Перед задачей читай только те CODEMAP-файлы, которые соответствуют области изменения. Если задача меняет код области, обнови соответствующий CODEMAP-файл в том же изменении.

## MCP

MCP нельзя использовать «на всякий случай». Если задачу можно решить по локальному коду и локальным docs, MCP не использовать.

Context7 MCP подключён только проектно через `.codex/config.toml`. Использовать Context7 только если нужна актуальная документация внешней библиотеки: добавляется библиотека, нужен актуальный API, есть ошибка версии, неясна сигнатура или локальной документации недостаточно.

Playwright MCP подключён только проектно через `.codex/config.toml`. Использовать Playwright только для UI/e2e-сценариев, скриншотов и проверки поведения интерфейса: если меняется UI, autocomplete popup, клавиатурное поведение, тема, пользовательский сценарий, нужен скриншот или воспроизведение UI-бага.

Не предлагать глобальный `codex mcp add` как основной способ настройки MCP для `ai-editor`. Его можно упоминать только как альтернативу для других проектов или личной глобальной конфигурации.

## Subagents

Использовать subagents для крупных feature-задач, независимого review, архитектурной проверки, сложного UI/e2e сценария или перед важным commit. Не использовать subagents для маленьких изменений, документации, переименования, форматирования и задач, где достаточно локальных проверок.

## Зависимости

Перед добавлением или установкой зависимости объяснить: зачем она нужна, почему нельзя решить проще, есть ли альтернатива в текущем стеке и как она влияет на поддержку проекта.

Запрет относится к лишним и необоснованным зависимостям. Необходимые npm/Rust/system dependencies для scaffold, запуска Tauri, сборки и тестов устанавливать можно, если они проверяемы и указаны в отчёте.

## Отчёты

Финальный отчёт должен содержать только: что изменено, какие проверки выполнены, что не удалось выполнить, какие риски остались и какие файлы затронуты.

## Пример context checklist

```text
Тип задачи: UI task
Читаю: AGENTS.md, docs/UI_DESIGN_GUIDE.md, docs/ARCHITECTURE.md
Не читаю: ROADMAP.md, ADR
MCP: не нужен
Subagents: не нужны
```
