# Стратегия тестирования

Тесты должны проверять реальное поведение редактора и не зависеть от установленной Ollama или другого локального inference backend, если task spec не требует интеграционную проверку.

## Unit tests

Покрывать чистую логику:

- фильтрация подсказок по prefix;
- выбор следующей/предыдущей подсказки;
- вставка выбранной подсказки;
- удаление дублей и ограничение количества;
- fallback при ошибке или пустом результате provider;
- функции ranking, parsing и state transitions.

Любые интеграции с локальным inference заменять тестовыми doubles в unit-тестах.

## Component tests

Проверять пользовательские сценарии:

- редактор отображается;
- popup появляется при вводе;
- стрелки меняют выбранный пункт;
- `Tab` вставляет подсказку;
- `Esc` закрывает popup;
- тема переключается;
- status bar показывает ожидаемые состояния.

## Feature verification

После feature-задачи tester должен проверять не только diff, но и реальное поведение приложения доступными средствами:

- запускать `npm run check`;
- запускать `npm run build`, если меняется scaffold, Tauri или сборка;
- запускать `npm run dev` или smoke/e2e-сценарий для UI, если окружение позволяет открыть приложение;
- сверять acceptance criteria из текущей задачи;
- явно фиксировать, что не было проверено и почему.

## Smoke/e2e

Минимальные сценарии: приложение стартует, редактор принимает ввод, подсказка вставляется, popup закрывается, fallback не ломает UI.

Playwright использовать для UI/e2e, скриншотов и проверки поведения интерфейса, когда локальных unit/component tests недостаточно.
