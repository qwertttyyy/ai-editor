# Стратегия тестирования

Тесты MVP 1 должны проверять реальное поведение редактора и не зависеть от установленной Ollama.

## Unit tests

Покрывать чистую autocomplete-логику:

- фильтрация подсказок по prefix;
- выбор следующей/предыдущей подсказки;
- вставка выбранной подсказки;
- удаление дублей и ограничение количества;
- fallback при ошибке или пустом результате provider.

Ollama не должна быть обязательна для unit-тестов. Любые интеграции с Ollama мокать.

## Component tests

Проверять пользовательские сценарии:

- редактор отображается;
- popup появляется при вводе;
- стрелки меняют выбранный пункт;
- `Tab` вставляет подсказку;
- `Esc` закрывает popup;
- тема переключается.

## Smoke/e2e позже

Smoke/e2e tests добавляются после scaffold, когда запуск приложения стабилен. Минимальные сценарии: приложение стартует, редактор принимает ввод, подсказка вставляется, fallback не ломает UI.

## Mock provider

`MockSuggestionProvider` обязателен для разработки MVP 1: он предсказуем, не требует сети и не зависит от Ollama.
